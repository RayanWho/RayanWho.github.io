# RAG（检索增强生成）

> **核心概念**：先查资料，再让 AI 基于资料回答。
>
> **全称**：Retrieval-Augmented Generation

---

## 🤔 为什么会有 RAG

普通大模型有两个问题：

### 问题 1：不知道你的私有内容

```
❌ 我公司的报销制度是什么？
❌ 这份产品文档里怎么定义"高价值用户"？
❌ 这份 PDF 的重点是什么？
```

这些内容通常**不在模型训练数据里**。

### 问题 2：容易"胡说"

模型有时候会一本正经地编，尤其是：
- 问它记不准的内容
- 问它没见过的信息
- 问企业内部资料

> 💡 **所以就有了 RAG**

---

## 🔄 RAG 是怎么工作的

RAG 一般分**两步**：

### 第一步：检索 🔍

先从外部知识库里找相关内容。

**知识库可以是：**
| 类型 | 示例 |
|------|------|
| 📄 文档 | PDF, Word, Markdown |
| 🌐 网页 | 网站内容 |
| 🗄️ 数据库 | 结构化数据 |
| 📁 文件 | 本地文件导入 |
| 📋 其他 | 公司文档, FAQ, 聊天记录 |

### 第二步：生成 ✍️

把检索到的内容连同你的问题一起发给大模型，让模型根据这些资料来回答。

> 💡 **所以它不是「纯靠脑子答」，而是：边查边答。**

---

## 📖 举个最容易懂的例子

比如你做一个**公司内部问答系统**。

**用户问：**
> 出差住宿报销上限是多少？

### ❌ 没有 RAG

模型可能凭训练记忆乱猜：
> 一般是 500 元一天……

但这**不一定是你公司的真实制度**。

### ✅ 有 RAG

系统先去知识库查到一段制度文档：
> 一线城市住宿报销上限为 800 元/天，二线城市为 500 元/天。

然后模型基于这段内容回答：
> 根据当前制度，一线城市住宿报销上限为 800 元/天，二线城市为 500 元/天。

**这就是 RAG。**

## 🧩 RAG 的核心组成

一个典型 RAG 系统，通常有这几个部分：

```
┌─────────────┐
│ 1. 文档加载  │ ← 把资料读进来
└──────┬──────┘
       ↓
┌─────────────┐
│ 2. 文本切块  │ ← 切成小段（300～800 字）
└──────┬──────┘
       ↓
┌─────────────┐
│ 3. 向量化    │ ← 用 Embedding 模型转成向量
└──────┬──────┘
       ↓
┌─────────────┐
│ 4. 向量数据库│ ← FAISS, Chroma, Milvus, pgvector, Weaviate
└──────┬──────┘
       ↓
┌─────────────┐
│ 5. 检索      │ ← 用户提问时找最相关文本块
└──────┬──────┘
       ↓
┌─────────────┐
│ 6. 拼接上下文│ ← "用户问题 + 检索内容"发给 LLM
└──────┬──────┘
       ↓
┌─────────────┐
│ 7. 生成答案  │ ← 模型基于上下文回答
└─────────────┘
```

### 各组件说明

| 组件 | 说明 | 常用工具 |
|------|------|----------|
| 📥 文档加载 | 把资料读进来 | PDF 解析，网页抓取，数据库读取 |
| ✂️ 文本切块 | 把长文档切成小段 | 按字数/标题/段落切分 |
| 🔢 向量化 | 文本转成向量 | Embedding 模型 |
| 🗄️ 向量数据库 | 存储和检索向量 | FAISS, Chroma, Milvus, pgvector, Weaviate |
| 🔍 检索 | 找最相关文本块 | 语义相似度搜索 |
| 📎 拼接上下文 | 问题 + 检索内容 | Prompt 组装 |
| ✍️ 生成答案 | 基于上下文回答 | LLM |

## 🆚 RAG 和微调有什么区别

很多人容易混淆这两者。

| 维度 | RAG | 微调 |
|------|-----|------|
| **方式** | 不重新训练，回答前先查资料 | 改模型参数，让模型学会某种能力 |
| **知识更新** | 快，文档改了重新入库即可 | 慢，需要重新训练 |
| **成本** | 低 | 高 |
| **适合场景** | 企业知识库问答 | 固定任务模式、风格学习 |

> 💡 **大多数知识库问答场景，优先是 RAG，不是微调。**

## 🎯 RAG 适合什么场景

RAG 非常适合：

| 场景 | 说明 |
|------|------|
| 🏢 企业知识库问答 | 公司内部制度、流程查询 |
| 🎧 客服 FAQ 问答 | 常见问题自动回答 |
| 📄 文档问答 | PDF/Word 内容查询 |
| ⚖️ 法律/医疗/金融 | 专业资料辅助查询 |
| 📚 技术文档助手 | API 文档、手册查询 |
| 📱 App 内功能 | "问文档"、"问手册"、"问合同" |

### 你可能感兴趣的方向

- AI 知识库问答 App
- 本地日志分析助手
- 电商合规文档问答
- 股票研究资料检索问答

## ✅ RAG 的优点

| 优点 | 说明 |
|------|------|
| 🎯 提高准确性 | 回答有依据，不容易乱编 |
| 🔒 能接私有知识 | 模型不知道你公司文档，但 RAG 可以查到 |
| 🔄 更新方便 | 文档更新后重新入库即可 |
| 📑 可追溯 | 可以把「参考片段」显示给用户 |

---

## ⚠️ RAG 的局限

RAG 不是万能的，也有问题：

| 局限 | 说明 |
|------|------|
| ❌ 检索错了，答案也会错 | 没找到真正相关的片段，模型容易答偏 |
| ✂️ 切块不好会影响效果 | 切得太碎→信息不完整；切得太大→检索不精准 |
| 🧠 多跳推理比较弱 | 需要跨多个文档综合推理时，简单 RAG 表现一般 |
| 🗑️ 依赖文档质量 | 垃圾文档进来，垃圾答案出去 |

## 📊 流程图理解

### 简化版

```
资料库 → 检索相关内容 → 把内容喂给模型 → 模型生成答案
```

### 展开版

```
文档 → 切块 → 向量化 → 向量库
                          ↓
用户提问 → 向量检索 → 找到相关片段 → 连同问题发给 LLM → 输出答案
```

## 📱 对 Android / App 开发者的意义

如果你做一个 AI App，RAG 通常是这样的分工：

### App 端负责

- 📤 上传文件
- 💬 输入问题
- 📺 展示回答
- 📑 展示引用来源
- 💬 会话管理

### 服务端负责

- 📄 文档解析
- 🔢 向量化
- 🔍 向量检索
- 📝 拼 Prompt
- 🤖 调 LLM
- 📤 返回结果

### 端侧 RAG（小规模）

也可以做，但通常规模较小：
- 本地少量文档
- 本地向量库
- 本地小模型 / 或云端生成
- 更适合隐私和离线场景

---

## 🎓 一句话总结

> **RAG = 先从知识库里检索相关资料，再让大模型基于资料生成答案。**

所以它不是让模型「背更多知识」，而是让模型在回答前**先开卷查资料**。

## 🔑 核心要点速记

| 要点 | 说明 |
|------|------|
| 🎯 核心思想 | 先检索，再生成 |
| 📚 知识库 | 私有数据、文档、FAQ |
| 🔍 检索方式 | 向量相似度搜索 |
| ✅ 优点 | 准确、可追溯、更新快 |
| ⚠️ 局限 | 检索质量依赖、多跳推理弱 |
| 🆚 vs 微调 | RAG 更灵活，成本更低 |
